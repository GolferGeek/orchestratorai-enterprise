# Technology Company Client Intake Checklist

**Document Type:** Client Intake Checklist — Technology Companies
**Prepared By:** OrchestratorAI Practice Management Consulting
**Version:** 1.0
**Last Updated:** April 2026
**Applicable To:** Software companies, SaaS providers, AI/ML firms, hardware manufacturers, IT consultancies, and technology startups seeking OrchestratorAI platform deployment or legal representation through OrchestratorAI-powered legal firms

---

## Purpose

This checklist governs the intake process for technology company clients. Technology companies present unique intake requirements due to their intellectual property portfolios, regulatory exposure across multiple jurisdictions, complex data handling obligations, and the technical sophistication required to evaluate their existing infrastructure for OrchestratorAI platform integration.

This document serves two audiences: (1) OrchestratorAI's own sales and onboarding teams evaluating a technology company as a prospective platform client, and (2) boutique legal firms using OrchestratorAI's knowledge base to onboard their own technology company clients.

---

## Section 1: Company Information and Structure

### 1.1 General Corporate Information

- [ ] Full legal name of entity
- [ ] State/jurisdiction of incorporation or organization
- [ ] Date of formation
- [ ] Entity type (C-Corp, S-Corp, LLC, LP, PBC, etc.)
- [ ] Federal EIN / Tax ID
- [ ] Registered agent name and address
- [ ] Principal place of business address
- [ ] All additional office locations (domestic and international)
- [ ] Company website URL(s)
- [ ] Primary contact name, title, email, and direct phone
- [ ] Secondary/backup contact name, title, email, and direct phone
- [ ] Authorized decision maker for legal/technology engagements (name and title)

### 1.2 Corporate Structure and Ownership

- [ ] Organizational chart (parent, subsidiaries, affiliates)
- [ ] List of all subsidiaries and their jurisdictions
- [ ] Ownership breakdown (founders, investors, employee equity pool)
- [ ] Cap table or summary of equity holders (if private)
- [ ] Board of directors composition
- [ ] Advisory board members (if any)
- [ ] Identify any government or foreign ownership interests
- [ ] Joint venture or strategic partnership entities
- [ ] Any pending or contemplated corporate transactions (M&A, restructuring, IPO)

### 1.3 Funding and Financial Overview

- [ ] Current funding stage (bootstrapped, seed, Series A-F, public)
- [ ] Total capital raised to date
- [ ] Most recent valuation (if available and willing to disclose)
- [ ] Current annual revenue range
- [ ] Primary revenue model (subscription, licensing, services, hardware, hybrid)
- [ ] Burn rate and runway (for pre-revenue or early-stage companies)
- [ ] Current banking relationships
- [ ] Outside accounting firm / CFO service

## Section 2: Products, Services, and Technology Stack

### 2.1 Product and Service Inventory

- [ ] Complete list of products and services offered
- [ ] Product architecture overview (monolith, microservices, serverless, hybrid)
- [ ] Primary programming languages and frameworks in use
- [ ] Deployment model for each product (on-premises, cloud, hybrid, edge)
- [ ] Cloud providers utilized (AWS, Azure, GCP, private cloud, multi-cloud)
- [ ] SaaS platform uptime SLA commitments
- [ ] Current monthly active users (MAU) or equivalent usage metrics
- [ ] API integrations offered to customers
- [ ] Third-party dependencies and vendor relationships critical to product delivery

### 2.2 Development Team Assessment

- [ ] Total engineering headcount (full-time, contract, offshore)
- [ ] Engineering team structure (teams, leads, reporting)
- [ ] Development methodology (Agile, Scrum, Kanban, etc.)
- [ ] Source control platform (GitHub, GitLab, Bitbucket, Azure DevOps)
- [ ] CI/CD pipeline tools and configuration
- [ ] Code review practices and approval requirements
- [ ] Testing practices (unit, integration, E2E, load, security)
- [ ] Average deployment frequency
- [ ] Incident response process and on-call rotation
- [ ] Technical debt assessment (self-reported severity: low/medium/high/critical)

### 2.3 Current AI/ML Capabilities

- [ ] Existing AI/ML models in production (list with brief descriptions)
- [ ] Model training infrastructure (local GPU, cloud GPU, managed services)
- [ ] Data pipeline architecture for model training
- [ ] Model versioning and experiment tracking tools (MLflow, W&B, etc.)
- [ ] AI governance policies in place (bias testing, fairness audits, explainability)
- [ ] LLM usage (OpenAI, Anthropic, open-source models, fine-tuned models)
- [ ] RAG (Retrieval-Augmented Generation) implementations
- [ ] AI-related customer-facing features
- [ ] AI safety and alignment practices
- [ ] Budget allocation for AI/ML initiatives (annual)

### 5.1 Data Classification

- [ ] Data classification policy (public, internal, confidential, restricted)

### 5.2 Privacy Compliance

- [ ] Privacy policy (current, published, reviewed by counsel?)

- [ ] CCPA/CPRA compliance status (if California exposure)
- [ ] Other state privacy law compliance (Colorado, Connecticut, Virginia, etc.)
- [ ] Cookie consent and tracking disclosure mechanisms
- [ ] Data Protection Officer (DPO) appointed? (name and contact)
- [ ] Data Protection Impact Assessments (DPIAs) completed

### 5.3 Data Security

- [ ] Information security policy document

- [ ] ISO 27001 certification status
- [ ] Penetration testing schedule and most recent results
- [ ] Vulnerability management program
- [ ] Encryption standards (at rest, in transit, key management)
- [ ] Access control and identity management systems
- [ ] Multi-factor authentication (MFA) implementation status
- [ ] Security incident response plan
- [ ] History of data breaches or security incidents (past 5 years)
- [ ] Cyber insurance coverage (carrier, limits, exclusions)

## Section 8: OrchestratorAI Platform Assessment

*This section applies when the technology company is being evaluated as a prospective OrchestratorAI platform client.*

### 8.1 Infrastructure Readiness

- [ ] Available hardware for local deployment:
  - [ ] Apple Mac Studio (M-series, 64GB+ RAM) — recommended for development teams
  - [ ] NVIDIA DGX Spark or equivalent GPU workstation — recommended for AI-heavy workloads
  - [ ] Server-class hardware with 64GB+ RAM for on-premises deployment
  - [ ] Willingness to procure recommended hardware if not available
- [ ] Network infrastructure assessment:
  - [ ] Internal network bandwidth (minimum 1Gbps recommended)
  - [ ] VPN or Tailscale mesh network capability
  - [ ] Firewall and network segmentation policies
  - [ ] DNS management capability
- [ ] Cloud infrastructure preferences:
  - [ ] Azure subscription (for Azure Foundry integration)
  - [ ] GCP project (for Vertex AI integration)
  - [ ] Hybrid deployment interest (local + cloud)

### 8.2 Use Case Assessment

- [ ] Primary use cases for OrchestratorAI platform:
  - [ ] AI-assisted code development (Compose agents)
  - [ ] Complex multi-step workflows (Forge LangGraph agents)
  - [ ] Internal process automation (Pulse automation)
  - [ ] External agent communication (Bridge/Gatekeeper)
  - [ ] Market prediction and analysis (Divinr.ai integration)
  - [ ] Document processing and RAG knowledge bases
  - [ ] Client-facing AI features
- [ ] Volume expectations:
  - [ ] Expected number of concurrent users
  - [ ] Expected monthly API call volume
  - [ ] Expected data volume for RAG knowledge bases
- [ ] Integration requirements:
  - [ ] Existing systems that must integrate with OrchestratorAI
  - [ ] API format preferences (REST, GraphQL, gRPC)
  - [ ] Authentication system integration (SSO, OIDC, SAML)

### 8.3 Security and Sovereignty Requirements

- [ ] Data sovereignty requirements (must data stay on-premises?)
- [ ] Sovereign mode deployment requirement (all AI inference local)
- [ ] Acceptable cloud regions for any cloud components
- [ ] Air-gapped deployment requirement
- [ ] Compliance frameworks that OrchestratorAI deployment must satisfy
- [ ] Security review or vendor assessment process to be completed
- [ ] Penetration testing requirements for deployed platform

### 8.4 Budget and Timeline

- [ ] Approved budget range for platform deployment
- [ ] Hardware procurement budget (if separate)
- [ ] Expected deployment timeline (target go-live date)
- [ ] Internal champion / project sponsor identified
- [ ] IT team resources allocated for deployment support
- [ ] Training budget for staff onboarding

## Notes

- Technology company intakes frequently uncover IP ownership ambiguities (especially around contractor-developed code and open-source contamination). Flag these early.
- AI/ML capabilities assessment is critical for OrchestratorAI platform fit. Companies with existing ML pipelines will have smoother deployments.
- Export control screening is mandatory for companies with international operations or foreign-national employees working on controlled technology.
- Data sovereignty discussions should happen during intake, not after deployment planning begins. Many technology companies have non-negotiable data residency requirements.
